# Claude Code: как провести одну сессию с максимальной отдачей

Стек и логика зафиксированы в `SPEC.md`. Этот файл — про **процесс** и готовые **промпты**.
Модель: **Sonnet 4.6**. Подход: spec-driven + plan mode + проектные skills + работа фазами с коммитами.

---

## A. Подготовка (2 минуты до старта)

1. Создай пустую папку, положи в неё `SPEC.md`, выполни `git init`.
2. Запусти `claude` в этой папке.
3. Переключись в **Plan Mode** (жми `Shift+Tab`, пока не появится Plan Mode). В нём Claude исследует и планирует, ничего не записывая, — ты согласуешь план до кода.
4. Держи под рукой команды: `/skills` (список скиллов), `/review`, `/simplify`, `/compact`, `/clear`, `/doctor`.

**Почему так выжимает максимум:**
- **Plan mode** не даёт модели сразу писать код «как поняла» — сначала план, который ты правишь.
- **CLAUDE.md** (создаём в фазе 0) автоматически грузится в каждый запуск — конвенции не теряются между `/clear`.
- **Проектные skills** в `.claude/skills/` грузятся по требованию: повторяющиеся правила (дизайн-токены, алгоритм слотов, паттерны компонентов) применяются единообразно и **не съедают контекст**, пока не нужны.
- **Коммиты после каждой фазы** = git как память. Между фазами можно `/compact` или `/clear` — состояние восстановится из git + CLAUDE.md + SPEC.md + skills. Качество ответов падает на длинных сессиях (после ~120k токенов внимание «плывёт»), поэтому компактимся проактивно.
- **Субагенты** для проверок/тестов работают параллельно и не засоряют основной контекст.

## B. Стартовый (мастер-)промпт

> Вставь это первым сообщением, **находясь в Plan Mode**. Не давай писать код, пока не утвердишь план.

```
Ты — senior full-stack инженер. Строим продакшн-MVP студии маникюра/педикюра
по спецификации в ./SPEC.md. Прочитай SPEC.md целиком — это единственный
источник правды, сверяйся с ним и не отклоняйся от стека и логики без явного
согласования со мной.

СНАЧАЛА (в plan mode, без кода) подготовь и покажи мне:
1. Пошаговый план реализации, разбитый на ФАЗЫ ровно в таком порядке:
   Фаза 0 — каркас: Next.js 15 + TS strict, Tailwind v4, shadcn/ui, Prisma+Neon,
            Auth.js v5, Zod, структура папок, CLAUDE.md, проектные skills.

            (Vitest) на все перечисленные кейсы. ЭТО ДЕЛАЕМ ДО UI бронирования.
   Фаза 4 — поток бронирования: /booking (4 шага), интерактивный пикер времени,

2. Какие ПРОЕКТНЫЕ SKILLS ты создашь в .claude/skills/ и зачем. Минимум:

3. Что положишь в CLAUDE.md: стек, команды (dev/build/test/db), конвенции,

4. Открытые вопросы ко мне, если что-то в SPEC неоднозначно.

Правила на всю сессию:

- Права и переходы статусов проверяются НА СЕРВЕРЕ, не только на клиенте.
- После КАЖДОЙ фазы: останавливайся, показывай diff/итог, делай git commit

- Не пиши заглушки вместо логики; если чего-то не хватает в SPEC — спроси.

```

После согласования плана выйди из Plan Mode (`Shift+Tab`) и скажи: `Поехали с Фазы 0. Сначала создай CLAUDE.md и skills, потом каркас.`

## C. Промпты по фазам (давать по одному, после «ок» к предыдущей)

Короткие — потому что вся тяжёлая часть уже в `SPEC.md`, `CLAUDE.md` и skills.

**Фаза 0 — каркас**
```
Фаза 0. Создай CLAUDE.md и все запланированные skills в .claude/skills/.
Затем подними каркас: Next.js 15 App Router + TS strict, Tailwind v4, shadcn/ui,
Prisma + Neon (env-переменные с заглушками и .env.example), Auth.js v5, Zod, Vitest,
структуру папок из SPEC. Проверь, что dev-сервер и тест-раннер стартуют. Commit.
```

**Фаза 1 — дизайн-система + лендинг**
```
Фаза 1. Активируй skill glaze-design-system. Реализуй токены, подключи Fraunces +
General Sans, базовые UI-компоненты и лендинг (hero с перламутровым mesh-градиентом,
услуги, мастера, галерея, отзывы, CTA). Signature — глянцевый пикер-«чип».
Сделай оркестрованную, сдержанную анимацию; reduced-motion и focus обязательно;
mobile-first. Покажи скриншоты до/после своей же критики. Commit.
```

**Фаза 2 — данные, auth, роли**
```
Фаза 2. Активируй prisma-conventions. Реализуй схему из раздела 3 SPEC, миграции,
сид (6 мастеров, ~12 услуг, графики, демо-записи и отзывы). Настрой Auth.js
(email+пароль и Google), сессии в БД, middleware доступа по ролям CLIENT/MASTER/ADMIN.
Тест: вход под каждой ролью ведёт в свой кабинет. Commit.
```

**Фаза 3 — ядро доступности (ключевая)**
```
Фаза 3. Активируй availability-engine. Реализуй getAvailableSlots как чистые
функции над интервалами (вход: график, отгулы, занятые записи, настройки;
выход: список стартов слотов) — отдельно от Prisma. Покрой Vitest всеми кейсами
из 4.2: смены с перерывом, отгул среди дня, буфер между записями, граница
сегодня+lead time, запрет через полночь, DST через date-fns-tz, «любой свободный
мастер». Запусти тесты субагентом, добейся зелёного. Commit.
```

**Фаза 4 — бронирование**
```
Фаза 4. Реализуй /booking в 4 шага (услуга → мастер/«любой» → дата+время → подтверждение).
Пикер времени дёргает доступность из Фазы 3. Создание записи PENDING — в транзакции
с pg_advisory_xact_lock по (masterId, date) и повторной проверкой слота; занятый
слот → 409 и просьба выбрать другое время. Анимированные переходы шагов. Commit.
```

**Фаза 5 — кабинет клиента**
```
Фаза 5. /account: список (ближайшие/прошедшие), детали записи, перенос и отмена
строго по правилам cancelCutoffH и переходам статусов, отзыв только для COMPLETED
(1 на запись, обновляет ratingAvg/ratingCount мастера). Все проверки на сервере. Commit.
```

**Фаза 6 — кабинет мастера**
```
Фаза 6. /master: входящие PENDING с подтверждением/отклонением (отклонение требует
причину → masterNote, клиент уведомляется), отметки COMPLETED/NO_SHOW, недельный/дневной
календарь, редактирование своих WorkingHours и TimeOff, экран отзывов и рейтинга.
Активируй server-action-patterns для мутаций. Commit.
```

**Фаза 7 — админка**
```
Фаза 7. /admin: дашборд с аналитикой (выручка, загрузка мастеров %, топ-услуги),
CRUD мастеров (включая их услуги и графики) и услуг, общий календарь всех мастеров,
модерация отзывов (isPublished), экран StudioSettings. Доступ только ADMIN. Commit.
```

**Фаза 8 — уведомления**
```
Фаза 8. Абстракция NotificationChannel (чтобы потом добавить SMS), реализация email
через Resend для событий из 4.3. Vercel Cron: напоминание за 24ч и просьба об отзыве
после COMPLETED. Шаблоны писем в тоне бренда. Commit.
```

**Фаза 9 — финальный проход**
```
Фаза 9. Прогон /review и /simplify по изменённым файлам субагентами. Проверь
доступность (focus, контраст, reduced-motion, клавиатура), адаптивность на мобайле,
отсутствие any на границах, что все мутации проверяют права и переходы статусов.
Напиши README (локальный запуск, env, миграции, сид) и краткий чек-лист деплоя
на Vercel + Neon. Финальный commit.
```

- **Восстановить контекст после `/clear`:**
  `Прочитай SPEC.md, CLAUDE.md и git log. Скажи, на какой мы фазе и что дальше.`
- **Ревью качества:**
  `Запусти субагента-ревьюера по файлам последней фазы: ищи дыры в правах, гонки, незакрытые переходы статусов, утечки any. Дай список с приоритетами, потом исправь критичные.`
- **Усилить дизайн, если вышло шаблонно:**
  `Сверь текущий UI со skill glaze-design-system. Где он скатился в дефолтный «AI-look»? Возьми один аккуратный риск ради айдентики и переделай, объясни что и почему.`
- **Контекст на исходе:**
  `Закоммить текущее, обнови CLAUDE.md разделом «состояние/что дальше», затем /compact.`

### Почему именно так используются «скиллы»

В Claude Code skill — это папка `.claude/skills/<имя>/SKILL.md` с YAML-frontmatter
(`name`, `description`) и инструкциями. Они грузятся **по требованию** (экономят контекст),
лежат в репозитории (одинаковы для всей команды) и дают **единообразие** на повторяющихся
задачах. Мы создаём четыре проектных скилла под самые рискованные на консистентность места
(дизайн, алгоритм слотов, конвенции БД, паттерны мутаций) и активируем нужный в начале
каждой фазы фразой «Активируй skill …». Плюс пользуемся встроенными командами `/review`,
`/simplify`, `/skills`. Это и есть «максимально использовать скиллы»: не один длинный промпт,
а переиспользуемые модули + spec + git, которые держат качество на всей сессии.
